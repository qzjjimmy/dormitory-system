# 智能宿舍分配算法 — 设计方案

> 日期：2026-05-29
> 状态：✅ 已确认

---

## 一、概述

为宿舍管理系统增加**基于多特征兼容性评分的智能宿舍分配算法**，覆盖两大场景：
- **新生批量分配**：入学时管理员一键分配，同班优先
- **调宿匹配推荐**：学生申请调宿时推荐兼容度最高的空床位

同时在登录页新增**学生自主注册**功能，注册时收集 10 项个人特征用于算法匹配。

## 二、数据库改造

`sys_user` 表新增 10 个特征字段（全部允许 NULL，兼容老数据）：

| 字段 | 类型 | 含义 | 取值 |
|------|------|------|------|
| `gender` | VARCHAR(10) | 性别 | 男/女 |
| `major_class` | VARCHAR(80) | 专业班级 | 如"软件工程2023级" |
| `sleep_habit` | VARCHAR(10) | 作息习惯 | 早睡/晚睡 |
| `smoking` | VARCHAR(10) | 是否抽烟 | 是/否 |
| `hobbies` | VARCHAR(200) | 兴趣爱好 | 逗号分隔，如"篮球,编程,音乐" |
| `cleanliness` | VARCHAR(10) | 卫生习惯 | 整洁/一般/随意 |
| `gaming` | VARCHAR(10) | 是否打游戏 | 是/否 |
| `snoring` | VARCHAR(10) | 是否打鼾 | 是/否 |
| `return_time` | VARCHAR(10) | 作息返回时间 | 早归/正常/晚归 |
| `noise_tolerance` | VARCHAR(10) | 噪音容忍度 | 安静/正常/热闹 |

**实现方式**：`DatabaseInitializer.java` 启动时 `ALTER TABLE ADD COLUMN IF NOT EXISTS` 逐个添加，兼容已有库。

MySQL 不支持 `ADD COLUMN IF NOT EXISTS`，改用捕获异常的方式：`try { ALTER TABLE ADD COLUMN } catch (SQLException) { /* 字段已存在 */ }`。

## 三、注册流程

### 3.1 前端改动

登录页底部增加 `还没有账号？立即注册` 链接，点击切换到注册表单。

注册分两页：
- **第一页（账号信息）**：用户名、密码、确认密码、真实姓名、手机号
- **第二页（特征信息）**：性别、专业班级、作息习惯、是否抽烟、兴趣爱好、卫生习惯、是否打游戏、是否打鼾、作息返回时间、噪音容忍度

注册角色默认 `student`，注册后 `room_no` 为 NULL（待分配）。

兴趣爱好预设选项（多选标签）：篮球、足球、羽毛球、跑步、游泳、健身、编程、阅读、音乐、电影、游戏、摄影、绘画、旅行、美食

### 3.2 后端改动

- `AuthController` 新增 `POST /api/auth/register`
- `UserService.create()` 扩展支持特征字段
- 校验规则：用户名唯一、必填字段不为空
- 注册成功返回格式同登录：`{ id, username, realName, role, phone, roomNo, token, profileComplete }`

## 四、老用户特征补全

### 4.1 强制补全弹窗（首次登录）

`UserService.login()` 返回值新增字段 `profileComplete`：
- 逻辑：10 个特征字段全不为 NULL → `true`，任一为 NULL → `false`
- 管理员和宿管角色固定返回 `true`（不需要特征）
- 前端检测到 `false` 时，遮罩弹窗展示特征表单（字段同注册第二页），填完提交 `PUT /api/user/profile` 后才进入主界面
- 弹窗不可关闭、不可跳过

### 4.2 个人设置可编辑

`AccountSettings.vue` 改造：特征字段从只读变为可编辑。学生角色显示特征编辑区，管理员和宿管不显示。

### 4.3 后端接口

新增 `PUT /api/user/profile`，请求体包含 10 个特征字段。仅学生角色可调用，根据 token 识别当前用户更新。

## 五、智能分配算法

### 5.1 兼容性评分模型

两两学生之间计算 0–100 的匹配度得分：

| 特征 | 层级 | 权重 | 匹配规则 |
|------|:--:|:----:|------|
| 性别 | 硬约束 | — | 不一致直接排除（得 0 分） |
| 专业班级 | 硬约束（阶段一） | — | 同班优先填满，阶段二跨班补位 |
| 作息习惯 | 软约束 | 25 | 一致满分，不一致 0 |
| 是否抽烟 | 软约束 | 20 | 一致满分，不一致 0 |
| 卫生习惯 | 软约束 | 15 | 一致满分，差一档 -7.5，差两档 0 |
| 是否打游戏 | 软约束 | 10 | 一致满分，不一致 0 |
| 是否打鼾 | 软约束 | 10 | 一致满分，不一致 0 |
| 作息返回时间 | 软约束 | 10 | 一致满分，差一档 -5 |
| 噪音容忍度 | 软约束 | 5 | 一致满分，差一档 -2.5 |
| 兴趣爱好 | 软约束 | 5 | 共享兴趣数 / 总兴趣数 比例计分 |

### 5.2 阶段一：同班优先分配

1. 待分配学生按 `major_class` 分组
2. 每个班级组内，按 4 人间逐间填满
3. 组内用兼容性评分排序，得分最高的 4 人分入同一间
4. 最后一间不满 4 人的，留出空床位进入阶段二

### 5.3 阶段二：跨班补位

1. 收集所有宿舍的剩余空床位
2. 对每个空床位，遍历剩余待分配学生
3. 计算该学生与目标宿舍现有成员的平均兼容度
4. 贪心选取最高分匹配填入
5. 更新该宿舍成员列表，重复直到全部分配完毕

### 5.4 调宿匹配推荐

1. 对申请调宿的学生，遍历所有有空床位的宿舍
2. 计算与该宿舍现有成员的兼容度
3. 返回 Top-3 推荐，附带兼容度得分和理由

时间复杂度 O(n × m)，n 为学生数，m 为宿舍数。

## 六、前端智能分配页面

角色：仅管理员可见。左侧菜单新增"智能分配"。

### 6.1 操作区（顶部）
- 显示待分配学生数、可用宿舍数、空床位数
- 按钮"开始新生分配"触发批量分配
- 分配完成后显示结果摘要

### 6.2 分配结果表（中间）
- 宿舍 × 床位矩阵表格，显示学生姓名和班级
- 鼠标悬停显示学生特征标签
- 每间宿舍旁显示兼容度均分

### 6.3 调宿推荐区（底部）
- 下拉选择待调宿学生
- 列出 Top-3 推荐空床位及兼容度得分
- 管理员确认后一键执行

## 八、兼容性约束

| 约束 | 规则 |
|------|------|
| 性别 | 同一宿舍内必须一致 |
| 专业班级 | 阶段一硬约束，阶段二无约束 |
| 宿舍上限 | 每间宿舍 4 人 |
| 老数据 | 特征字段均为 NULL，标记 `profileComplete=false` |
